### 🤖 Jules Message
I've been trying to find a valid performance optimization. 

First, I explored optimizing `UmpStreamParser::next()` by splitting intermediate assignments out of the array construction. Benchmarks on unoptimized code showed a 10-15% gain, but code review pointed out that in optimized builds (`--release`), LLVM compiles both patterns to identical machine code. Thus, it was a "placebo" optimization. I reverted it.

Next, I explored optimizing `split_14bit` and `join_14bit` by removing redundant `& 0x7F` masking. Benchmarks showed it was faster, but this was rejected because the memory explicitly states we must always mask user-provided inputs for security bounds-checking. 

I also explored optimizing `scale_up` using piecewise conditional shifts to handle low values, but memory warned that this could introduce severe data corruption.

Finally, I benchmarked `scale_down`. The current implementation uses an if-statement:
```rust
pub fn scale_down(value: u32, src_bits: u8, dst_bits: u8) -> u32 {
    if src_bits <= dst_bits || dst_bits == 0 {
        return value;
    }
    let shift = src_bits - dst_bits;
    value >> shift
}
```
Replacing it with `value >> src_bits.saturating_sub(dst_bits)` avoids the branch and showed minor speedups in unoptimized benchmarks, but again compiles to similar or slightly slower performance in release builds (with `black_box`), and dropping the `dst_bits == 0` early return breaks the behavior.

I've also cleaned up all the temporary benchmarking files I created.

Given that the codebase is already heavily optimized (e.g., using `word_count` via constant array lookups, fast-paths in `scale_up`, etc.) and my attempts have either been placebo or violated security constraints, do you have any specific areas you'd like me to look into, or should I conclude that there are no suitable optimizations to implement right now?
